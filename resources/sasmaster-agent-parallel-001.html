01 — Problem Statement
Current SaSMaster build loop is sequential: one Claude Code session, one agent context, one task at a time. For complex multi-component builds — e.g. a Research Portal requiring scraper + API + UI + QA — this creates bottlenecks. The RESEARCH-PORTAL-001 milestone is the first natural candidate for parallelism.
Sequential today → Parallel fan-out → RESEARCH-PORTAL-001

03 — Recommended Architecture for RESEARCH-PORTAL-001
Use Option C (Claude Code Task tool) for the build, Option D (async queue) for the recurring data pipeline.

04 — Inter-Agent Communication Patterns
PATTERN 01
Return Value Passing
Sub-agent returns result as text → orchestrator parses → passes to next agent.
BEST FOR
Research results, design specs, file paths
Limit: output truncated ~10K tokens
PATTERN 02
S3 File Handoff
Agent A writes to S3 agent-handoff path. Agent B reads from same path via shared key.
s3://sasmaster-2026/ agent-handoff/ {session_id}/ {artifact}.json
BEST FOR
Large artifacts: schemas, full HTML files, data samples
Pattern: already used for jobs/queue.json
PATTERN 03
Shared File System
Agent writes to ~/SaSMaster/tmp/, orchestrator or next agent reads from same path.
BEST FOR
Local builds where agents share the same Mac filesystem
Not available in Railway/cloud context
05 — Failure Handling Rules
1
Partial failure is not total failure
If 1 of 3 parallel agents fails, collect the 2 successes and retry the failed one with error context.
2
Never auto-retry more than once
Log the failure to DONE_LOG.md and surface to Shiv. No silent retry loops.
3
QA is always the final gate
No fan-out result is accepted without QA agent sign-off. QA is never parallelized.
4
Cost guard applies per sub-agent
Each sub-agent session is independently token-budgeted (alert 150K, abort 250K via HARDEN-001).

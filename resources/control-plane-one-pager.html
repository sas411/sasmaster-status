The Control Plane, Explained

build-discipline §15
2026-06-07

Your platform has two kinds of code. One kind does the work — scrapes data, fuses keys, renders screens. The other kind decides when, where, and whether the work runs at all. That second kind is the control plane. It's small, it's quiet, and when it breaks, nothing screams — the platform just stops.

🗼
The Control Plane = the air-traffic tower

It flies no planes. It dispatches, schedules, routes, and restarts. If the tower goes dark, every plane is fine in isolation — but nothing takes off, and no one is told.

✈️
The Workload = the planes

Scrapers, fusion jobs, the portal UI. If one plane has a problem, it fails loudly — an error, a red build, a missing file. You see it immediately and fix that one thing.

It's control plane if…

touches HOW the system runs
  • It drains or feeds the command queue
  • It schedules anything (cron, launchd, timers)
  • It routes commands or emits heartbeats
  • It selects a worker or fails over to backup
  • It auto-restarts a daemon
  • A bad value makes it silently stop dispatching

It's just workload if…

touches WHAT the system produces
  • It scrapes, fuses, or transforms data
  • It renders a screen or component
  • It edits a document or skill
  • A bug shows up as a visible error or red build
  • You can fix it without the whole platform stalling
  • It's downstream bookkeeping (logs, boards)

⚠ Why it gets special treatment: it fails silently

A broken scraper throws an error you see in minutes. A broken control plane throws nothing — it just quietly stops dispatching, and you find out hours later from a queue that's climbing instead of draining. That asymmetry is the whole reason for the §15 gate: the cost of a quiet control-plane bug is measured in hours of dead platform, not a single red build.

CASE — JARVIS-QUEUE-001 (Jun 7): one hardcoded-credential line made the queue drainer fail without error. Queue sat stuck ~5.5 hrs. Nothing alerted. The fix was one line; the cost was a morning.
The rule: every control-plane change clears this gate first
01
Impact

What depends on this? What breaks if it stalls or returns a wrong value?

02
Blast radius

Does it touch queue / heartbeat / worker / restart? If yes → highest scrutiny.

03
Rollback

State the one-step revert before shipping. Can't name it → don't ship.

04
Non-blocking

No sync I/O in a handler. Timeouts exist. Can't hang the loop.

05
Secret / env

New env var actually SET in the target env? Missing secret = not shippable.

06
Post-change health

Run !status. Queue draining to 0? Not green = not done — revert.

The 5-second test
Does this change dispatch, schedule, route, restart, or drain?
YES → it's control plane → run the §15 gate  ·  NO → it's workload → ship normally

Credential transport

CSV → named profile → shell substitution · clipboard-free · LLM-context-free

5-step canonical flow · clipboard avoided · LLM context never sees the secret value

# After downloading CSV from AWS Console aws configure --profile sasmaster-ci # CRITICAL gate — test locally before pushing to GH aws s3 ls s3://sasmaster-2026/knowledge-bank/ --profile sasmaster-ci # Shell substitution → value never appears in command history or LLM context gh secret set AWS_ACCESS_KEY_ID --body "$(aws configure get aws_access_key_id --profile sasmaster-ci)" --repo sas411/sasmaster gh secret set AWS_SECRET_ACCESS_KEY --body "$(aws configure get aws_secret_access_key --profile sasmaster-ci)" --repo sas411/sasmaster # Clean up rm ~/Downloads/sasmaster-ci-key*.csv

07

Hard-won lessons

7 principles · each earned by a specific failure on May 23 · encode now, save future hours

Lesson 01

Credential transport rule

Never pbpaste-pipe AWS secrets to gh secret set — silent paste corruption is real. Always: aws configure --profile + shell substitution.

Lesson 02

Local verification before CI

Test credentials with aws s3 ls --profile X BEFORE pushing to GitHub Actions. Isolates "is it the cred?" from "is it the workflow?" — saves hours of cycle-time on debugging.

Lesson 03

Secrets never touch LLM context

Native Terminal only for raw secret values. Both Claude Code session logs and Claude.ai transcripts retain history. File paths are fine; file contents containing secrets are not.

Lesson 04

JARVIS scope is tracking, not relay

Multi-phase task handoffs go via Claude Code terminal stdin, NOT via Slack signals. JARVIS handles Kanban (!task, !done, !status) — not stdin injection.

Lesson 05

Two-workflow CI architecture

Reactive (push-triggered) + Proactive (scheduled) is the right pattern. Single-workflow approaches go stale between cron windows OR miss drift detection.

Lesson 06

Memory governance discipline

Hot tier (30 slots) reserved for working style + invariants + active commitments + evergreen lessons. Sprint completions and tactical detail age to Obsidian within ~14 days.

Lesson 07

AWS root account hygiene

Never store root password in plain-text Notes. Rotate immediately on exposure. MFA always active. Zero programmatic access keys on root — use scoped IAM users.

08

Runbook excerpts

Common ops · copy-pasteable command sequences

🔄 Rotate CI access key

  1. AWS Console → IAM → user → create new access key
  2. Download CSV
  3. aws configure --profile sasmaster-ci
  4. Local test: aws s3 ls --profile sasmaster-ci
  5. Push to GH secrets via shell substitution
  6. Trigger workflow, watch green
  7. Deactivate old key after 1-2 green runs

📌 Add new hot-tier memory

  1. Audit current 30 slots, identify stale tactical
  2. Draft Obsidian archive entry first (preserves verbatim text)
  3. Replace slot via memory_user_edits
  4. Trigger MEMORY-HYGIENE task to write archived entry

🩺 Diagnose failed CI sync

  1. gh run view <run_id> --log-failed
  2. If auth error: test locally with --profile sasmaster-ci
  3. If local works but CI fails: GH secret is stale → re-push
  4. If local fails: IAM policy issue → check scope/region

🚨 Respond to exposed secret

  1. Rotate the credential IMMEDIATELY
  2. Delete the exposure source (note, screenshot, log)
  3. Confirm MFA still active on parent account
  4. Check CloudTrail for suspicious activity
  5. Document in DONE_LOG with remediation steps

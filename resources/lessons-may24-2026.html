Architecture Decisions
🏗
JARVIS: Socket Mode → HTTP Events API (not the other way around)
Socket Mode keeps a persistent WebSocket from Mac → Slack. HTTP Events API is Slack → Railway (push). The architectural reason to flip: Railway is always-on cloud; Mac is not. Never require the Mac to be awake for inbound events.

Rule: Cloud-hosted service (Railway) must own the inbound path. Mac-side is for outbound (scraping, builds). launchd daemon for JARVIS was wrong architecture from the start.

#jarvis#railway#infra

⚡
DuckDB on Railway: use S3 pre-cache, not live queries
DuckDB httpfs extension + AWS CLI not available in Railway Nixpacks. Even with ListObjectsV2Command SDK, extension install fails silently. Correct pattern: Mac cron runs heavy DuckDB jobs, writes JSON cache to S3, Railway reads cache.

Architecture: Mac-side s3-entity-counts.js → sync-to-s3-cache.sh → cache/api/data-counts.json → Railway readS3JSON()

Anti-pattern: Trying to run DuckDB extension installs in container builds. Never works without a custom Dockerfile.

#duckdb#railway#s3

🔒
SSE 30-second Railway timeout: replace with async job + poll
Railway kills SSE connections after 30s. Dr. Scoop chat used SSE → unpredictably cut off mid-answer. Solution: POST to /api/jobs (202 accepted), GET poll every 30s, result delivered on completion. No timeout risk.

Rule: Never use SSE for responses that might take >25s on Railway. Use the async job queue pattern from HARDEN-006.

#sse#railway#api

Security Patterns
🚨
INTERNAL_SERVICE_TOKEN drift: .env vs Railway env can diverge silently
Railway env vars set via dashboard don't auto-sync to .env. Token 7fa907b was set in Railway but .env still had the old value. /api/* calls from Mac-side scripts were silently returning 401 with no clear error message.

Fix: After setting Railway env vars, always run vercel env pull .env.local or manually update .env to match.

Prevention: INFRA-DB-002 will add a health endpoint that reports token fingerprint without exposing the full token.

#security#env#railway

🔍
xapp- tokens in Slack SDK: deprecated and source of credential drift
Slack xapp- tokens are per-app-instance tokens from Socket Mode. After migrating to HTTP Events API, the xapp- token is no longer needed but was still being injected. Dropped from .env to reduce credential surface area.

Rule: Audit and remove credentials immediately when they're no longer in the hot path. They accumulate silently and create compliance risk.

#slack#credentials#security

Agent & Build Patterns
🤖
Domain-split (not layer-split) is the correct fan-out primitive
Layer-split (scraper agent → API agent → UI agent) creates sequential dependencies at every boundary. Domain-split (Ad Intel agent owns its full scraper+API+UI vertical) enables true parallelism. Each domain agent is self-contained and stateless relative to peer domains.

Cost model: ~$0.55 parallel vs $0.25 sequential for a 4-section portal build. Speed gain ~40% wall-clock. Worth it when tasks are genuinely independent.

Decision rule: Only fan-out when tasks don't share mutable state. If agent B reads agent A's output, keep them sequential.

#agents#parallel#architecture

⚡
Backtick escaping in heredoc Python → bash substitution
In process-jobs.sh, the Python heredoc was generating lines with `{sha}` — bash tried to execute {sha} as a command (non-fatal but produced stderr noise). Required escaping as \`{sha}\` inside the Python f-string.

Rule: Any backtick inside a Python heredoc embedded in bash must be escaped with \`. This applies to any `code` Markdown formatting in dynamically generated text.

#bash#python#scripting

📊
Schema drift detection: 3 S3 datasets, suspect last write, exit codes for cron
schema-drift-check.py reads the last 5 S3 batches per dataset, compares column names and types. If a new column appears or type changes, it flags the write as suspect and exits non-zero. Daily cron runs this; Slack alert fires on non-zero exit.

Pattern: Schema drift is a silent bug. The check must run daily and alert loudly — not weekly, not quietly. The cron exit code is the gate; never swallow failures.

#schema#s3#data-quality

Portal Design
🎨
B2C portal needs distinct visual identity from War Room (the internal tool)
War Room is lavender/purple — internal operations aesthetic. Research Portal is amber/terminal black — B2C product, premium media buyer experience. They serve different audiences and must not look related.

Rule: Every new portal starts with a design brief that explicitly names the target audience and differentiates it from existing portals. Never clone War Room's palette for B2C.

#ui#portal#b2c

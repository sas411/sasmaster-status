Layer 1

DISTRIBUTED
MEMORY

SaSMaster operates across four interlocking memory systems — each serving a distinct timescale and purpose. Claude.ai Memories, CLAUDE_MEMORY.md, and TASKS.md are live. The Obsidian vault mirrors operational state for human-readable handover — sync wired via sync-obsidian.sh, task queued.

✦
Claude.ai Memories
auto-generated · claude.ai
Persistent semantic memories derived from every conversation. Captures Shiv's working style, project context, sprint priorities, product vision, personal preferences, and long-term strategic context. Injected automatically into every new session — the system's "long-term memory."
Strategy Layer Auto-Updated Cross-Session Semantic
# What it holds
→ SaSMaster system state
→ Sprint priorities & blockers
→ Design tokens & canonical files
→ Division of labor rules
→ Recent wins & decisions
▤
CLAUDE_MEMORY.md
~/SaSMaster/CLAUDE_MEMORY.md
The operational ground truth for Claude Code sessions. Loaded as the very first command in every terminal session — before any code is written. Contains architecture state, S3 paths, known issues, recent builds, and what's live vs planned. Updated by update-memory.sh.
Execution Layer File-Based Per-Session Load Always Current
# Sections
Current Sprint — active priorities
Architecture State — live vs planned
Recent Wins — last 7 days
Known Issues — bugs / blockers
S3 State — latest data dates
☰
TASKS.md
~/SaSMaster/TASKS.md
The self-scoring task queue. JARVIS checks this at 7AM and 2PM EDT — if HIGH-priority items exist, it automatically fires a build. Tasks flow from JARVIS commands, Claude.ai sessions, and async build outputs. Tagged with #skill-worthy, #ui, #data, #infra.
Task Queue Self-Scoring JARVIS Trigger 7AM + 2PM

AUTONOMOUS &
SELF-LEARNING

The system doesn't just execute — it observes its own outputs, captures successful patterns as new skills, and improves with every build cycle. JARVIS's self-scoring queue closes the loop from task to learning.
